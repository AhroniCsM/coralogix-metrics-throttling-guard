Under the hood

Architecture

Two small shell scripts (bash + grpcurl) on any scheduled host. They sit beside your stack, not in your data path — your metrics keep flowing normally.

What it actually does

All usage is evaluated per UTC calendar day — the same boundary Coralogix resets on.

Detector — every 15–60 min

  1. Pages through every metric with GetMetricUsages for today and keeps those over THRESHOLD_UNITS on the chosen dimension.
  2. Calls List and drops any metric already blocked — never double-blocks or claims another rule.
  3. Applies your allowlist and per-run cap to the remaining candidates.
  4. Calls Block in chunks, then re-Lists to confirm and records each new rule (name + rule id) in a date-bucketed state.json.

Unblock — hourly, acts at 00:00 UTC

  1. Wakes hourly and exits immediately unless the current UTC hour is the configured unblock hour.
  2. For every past-day bucket, gathers the rule ids it recorded as its own.
  3. Calls Allow by rule id only, in chunks, removing each id from state as it succeeds.
  4. Deletes a day bucket only once fully unblocked; a failed unblock is kept and retried, never lost.

State is written atomically and both scripts share a file lock, so they can't race or corrupt state around the midnight boundary. The whole thing is idempotent and crash-safe.

The three gRPC calls

Detect

UsageService.GetMetricUsages

Reads per-metric daily usage and flags anything over your threshold. Read-only — also used by the healthcheck and preview run.

Block

MetricsBlockingRules.Block

Creates a blocking rule that stops the metric's ingestion for the rest of the UTC day. Returns a rule id the tool records as its own.

Unblock

MetricsBlockingRules.Allow

Removes rules by id — never by name — so it can't touch a rule you created by hand. Runs right after the daily reset.

What each script does

Two scripts do the real work; the rest are a shared library, a safe test runner, and scheduling.

detect_over_threshold.sh scheduled · 15–60 min
The detector. Pages through every metric's daily usage, keeps those over your threshold, then (if enforcement is on) skips already-blocked metrics, applies your allowlist + per-run cap, and blocks the rest — recording each new rule (name + id) in state.json. With enforcement off it just detects and logs.
unblock_midnight.sh scheduled · acts 00:00 UTC
The unblock job. Wakes hourly, acts only during the configured UTC hour, and unblocks everything it blocked on completed days — by rule id only, in chunks, removing each from state as it succeeds. A failed unblock is kept and retried, never lost.
lib_common.sh shared library
Sourced by both scripts. Provides logging, the file lock, atomic state writes, the auth header, the grpcurl wrappers with retry, and the read-only healthcheck. Not run directly.
run_test.sh run by hand
Safe staged runner — healthcheck · dryrun · live-one <metric> · unblock-now · status. Encodes the safe presets so you can't accidentally widen the blast radius. Never edits your config.
install_launchd.sh + *.plist macOS scheduling
Renders the schedule from your config and loads it via launchd: install · uninstall · run · status · doctor. On Linux you use cron / CronJobs instead.
config.env the only file you edit
Every setting — API key, region, threshold, which dimension, enforcement switches, schedule, notifications.

How to deploy it

Read-only until you explicitly enable enforcement. Prove the round-trip by hand, then schedule it.

1 · Prerequisites

# the runner just needs three CLIs on PATH
grpcurl   # gRPC client (talks to Coralogix over reflection — no protos)
jq        # JSON processing
flock     # cross-process lock (ships with util-linux; brew install flock on macOS)

2 · Configure (config.env — the only file you edit)

CX_API_KEY="<team-or-personal-api-key>"   # DataAnalytics read + manage blocking rules
USAGE_ENDPOINT="api.<region>.coralogix.com:443"  # eu1 eu2 us1 us2 ap1 ap2 ap3
USAGE_FIELD="cardinality"      # or unit_usage | bytes_volume | sample_count
THRESHOLD_UNITS="100000"       # flag metrics over this, per day
BLOCK_ENABLED="false"          # master switch — start in preview (blocks nothing)

3 · Prove it by hand (read-only → one real block → unblock)

./run_test.sh healthcheck          # verify key + endpoint + auth on both APIs
./run_test.sh dryrun               # real detection, blocks NOTHING
./run_test.sh live-one <metric>     # block exactly one metric (asks to confirm)
./run_test.sh unblock-now          # lift it again — full lifecycle proven

4 · Schedule it (production)

# Linux cron — detector every 15 min, unblock hourly (self-gates to 00:00 UTC)
*/15 * * * *  /opt/metrics-guard/detect_over_threshold.sh  >> /var/log/metrics-guard.log 2>&1
5   * * * *  /opt/metrics-guard/unblock_midnight.sh        >> /var/log/metrics-guard.log 2>&1
# …or two Kubernetes CronJobs (config in a Secret), image = your scripts + grpcurl + jq
kind: CronJob   # metrics-guard-detect
  schedule: "*/15 * * * *"
  command: ["/app/detect_over_threshold.sh"]
  envFrom: [{ secretRef: { name: metrics-guard-config }}]
---
kind: CronJob   # metrics-guard-unblock
  schedule: "5 * * * *"
  command: ["/app/unblock_midnight.sh"]

macOS hosts can use the bundled install_launchd.sh instead of cron. Run one folder (config + state) per account/region; guard cardinality and volume as two separate instances.
